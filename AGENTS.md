# AGENTS Instruction

The main developer documentation for Wagtail lives in the `docs/contributing` directory. Here is additional guidance for agents.

## Pull request guidelines

- Describe the "why" of the changes, why the proposed solution is the right one.
- Highlight areas of the proposed changes that require careful review.
- Always add a disclaimer to the PR description mentioning how AI agents are involved with the contribution.
- Do not add commits that are unrelated to the purpose of the PR. Check the commit history against the latest upstream main branch before pushing to ensure that only relevant commits are included in the PR.

### AI/Agent disclosure template (copy/paste & edit):

**If human review has *not yet* occurred (use this initially):**
> This pull request includes code written with the assistance of AI.  
> The code has **not yet been reviewed** by a human.

## Wagtail-specific pitfalls for AI agents

### StreamField and StreamBlock template access

- Wagtail's StreamBlock and StreamField use the same template syntax, but they differ from standard Django field access. When you use StreamField and other custom block types in Wagtail templates, you usually need to use the value property in your data variables.

### Django migrations

- Do **not** hand-write or hand-edit files under any `migrations/` directory. Migrations are generated by Django's `makemigrations` command from changes to `models.py`. If a change touches a model, run the command and commit the resulting migration file verbatim, without manual edits to its `operations = [...]` list. Editing a migration by hand is a frequent failure mode for AI-led contributions and is almost always wrong.
- Do not create migrations that reference fields, models, or apps that do not exist on the branch — Django will fail to apply them.
- Do not delete existing migration files to "tidy up" history; downstream projects have already applied them.

### Translations and `.po` files

- Translations live in `wagtail/locale/<lang>/LC_MESSAGES/django.po`, but those files are managed through [Transifex](https://explore.transifex.com/torchbox/wagtail/), not via pull requests. Do not edit `.po`, `.pot`, or compiled `.mo` files directly in a PR — the next sync from Transifex will overwrite them. Submit translation changes through Transifex instead.
- It is fine, and often required, to mark new user-facing strings for translation in Python and templates using `django.utils.translation.gettext` / the `{% translate %}` template tag — that part is in scope for PRs.

### British English for user-facing text

- User-facing strings in Wagtail use British English (for example, "behaviour" not "behavior", "colour" not "color"). Identifiers and code symbols use American English when needed to match Django / browser APIs (for example, CSS `color`). See the [language section of the general guidelines](https://docs.wagtail.org/en/latest/contributing/general_guidelines.html#language) for the full rule.

### Honest disclosure of testing

- The [AI disclosure template](#ai-agent-disclosure-template-copy-paste--edit) above is intentionally worded to say the code "has not yet been reviewed by a human". Do **not** edit it (or the PR body) to claim manual testing, browser testing, or real-world experience that the agent did not actually perform — making false statements of that kind is explicitly listed as an unacceptable use of generative AI in Wagtail's [contributor guidelines](https://docs.wagtail.org/en/latest/contributing/general_guidelines.html#use-of-generative-ai).
